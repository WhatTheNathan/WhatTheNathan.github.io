Swift Style Guide自用笔记

闲谈

我的Swift风格基本上是跟着白胡子老爷爷，然后被一些开源项目的源码影响，这次阅读了Google的Swift Style Guide，还是发现了不少自己风格上的问题，这里的笔记仅仅是针对自己所记录的笔记。

Source File Basic

  • 对一个类型的单个protocol的遵循，应该命名为来MyType+protocol.swift
  • 对一个类型的多个protocol的的遵循，可以命名为MyType+Additions.swift

Source File Structure

  • 更推荐import完整的模块而不是import子模块，除非import整个模块会污染top-level的命名空间，以及必须import子模块才能达到目的。
  • import statements应分组按字母首字母顺序编写

Type, Variable, and Function Declarations

  • 通常来说，一个文件只contain一个top-level类型，除非
    • 一个类与其delegate的protocol
    • 一个类型与其相关的helper类型，这种情况下，helper类型声明为fileprivate
  • 一个文件中类型、变量、函数的声明顺序对代码可读性有极大的影响。应使用// Mark: -来分组
  • 当一个类型有多个initialzers或subscripts，或者文件/类型中有多个函数有一样的base name，不一样的argument Labels，它们应该顺序出现，且无间隔。
  • 可以使用Extensions来组织将一个类型的代码按功能分类。

One Statement Per Line

  • 每一行一句statement，除非结束处包括一个含零或一句statement的block
  • 包含重要函数调用时，应该写成multi-line statement

Line-Wrapping

  • ,相隔的list要么horizontally要么vertically
  • 花括号{应该在同一行，除非如该例子，其所在行缩进了+2，其需要另起一行。

Formatting Specific Constructs

Non-Documentation Comments

  • 非文档注释不要使用C-style的/*...*/，应该使用//

Trailing Closures

  • 若函数的重载仅仅是argument label不同时，会在trailing closure产生歧义
  • 多个closure参数时，不应使用trailing closure

Naming

Delegate Methods

  • Delegate Methods应使用source object为第一个形参

  • 对于只使用source object为参数的delegate method

    • If return Void，即用于通知delegate事件的发生，应命名为delegate's source type + indicative verb phrase，参数为unlabeled。

    • If return Bool，即用于设置断言，应命名为delegate's source type +indicative or conditional verb phrase，参数为unlabeled。

    • If return 其他类型的值，比如查询delegate’s source type的infomation，应命名为描述查询的属性的名词短语。参数应该labeled为介词或带介词的短语，并且与delegate’s source type结合成名词短语。

  • 对于包括额外参数的delegate method，函数的base name即为delegate’s source type，其为unlabeled。

    • If return Void，第二个参数应labeled为an indicative verb phrase描述事件。

    • If return Bool，第二个参数应labeled为an indicative or conditional verb phrase描述基于参数的返回值。

    • If return 其他类型的值，第二个参数labeled为a noun phrase and trailing preposition描述基于参数的返回值。

Programming Practices

Types with Shorthand Names

  • Void是()的typealias，使用() -> Void 而不是 () -> ()

Optional Types

  • 使用Optional充当哨兵的返回值

Error Types

  • 除非error的结果是无可恢复的状态，导致程序的立即终止，否则不要使用try!，应该使用do-catch 或者 try?

Force Unwrapping and Force Casts

  • 除非非常确信，否则不要使用强制解包，使用强制解包应加以注释说明

Nesting and Namespacing

  • 声明无case的enum是定义一组相关声明的规范方法

fallthrough in switch Statements

  • 当switch的多个case执行一样的statement时，case的pattern可以combine为ranges

Pattern Matching

  • pattern matching中无let前缀是用于匹配，有let为binding

Playground Literals

  • 在non-playground production code中应禁止使用The graphically-rendered playground literals #colorLiteral(...), #imageLiteral(...), and #fileLiteral(...)

Documentation Comments
